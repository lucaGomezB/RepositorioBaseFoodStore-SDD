# Skills Map

Mapa de las skills disponibles con su trigger/contexto y descripción corta.

| Skill | Trigger / Contexto | Descripción corta |
|---|---|---|
| `branch-pr` | When creating a pull request, opening a PR, or preparing changes for review. | PR creation workflow for Agent Teams Lite following the issue-first enforcement system. |
| `code-review-excellence` | When reviewing pull requests or establishing review standards. | Master effective code review practices to provide constructive feedback, catch bugs early, and foster knowledge sharing while maintaining team morale. |
| `dashboard-crud-page` | When creating any CRUD page in the Dashboard sub-project (`src/pages/`). | Standardizes Dashboard CRUD pages with consistent structure, hooks, and UI patterns. |
| `fastapi-templates` | `.py / fastapi-python / backend API scaffolding.` | Create production-ready FastAPI projects with async patterns, dependency injection, and comprehensive error handling. |
| `find-skills` | When the user asks to discover or install a skill. | Helps users discover and install agent skills when they ask for functionality that might exist as an installable skill. |
| `go-testing` | When writing Go tests, using teatest, or adding test coverage. | Go testing patterns for Gentleman.Dots, including Bubbletea TUI testing. |
| `issue-creation` | When creating a GitHub issue, reporting a bug, or requesting a feature. | Issue creation workflow for Agent Teams Lite following the issue-first enforcement system. |
| `judgment-day` | When user says "judgment day", "judgment-day", "review adversarial", "dual review". | Parallel adversarial review protocol with two independent blind judge sub-agents, synthesis, fixes, and re-judging. |
| `kb-creator` | When user asks to create, build, or generate a knowledge base for a project. | Builds a structured project knowledge base of canonical markdown files from source documents or interactive creation. |
| `openspec-apply-change` | When implementing an OpenSpec change. | Implement tasks from an OpenSpec change and continue implementation work. |
| `openspec-archive-change` | When archiving a completed OpenSpec change. | Archive a completed change in the experimental OpenSpec workflow. |
| `openspec-design` | When the orchestrator or user needs a design document created or updated for a change. | Create a technical design document for a change using the openspec CLI. |
| `openspec-explore` | When the user wants to explore ideas or clarify requirements before a change. | Enter explore mode to investigate problems, clarify requirements, and think through options. |
| `openspec-init` | When user wants to initialize OPSX in a project, or says "opsx init", "iniciar opsx", "openspec init". | Initialize OPSX in a project and scaffold the `openspec/` directory and config. |
| `openspec-onboard` | When the user wants to learn OPSX or do a guided first change. | Guided walkthrough of the OPSX workflow using the real codebase. |
| `openspec-propose` | When proposing a new OpenSpec change. | Propose a new change with all artifacts generated in one step. |
| `openspec-spec` | When the orchestrator or user needs specs created or updated for a change. | Write or update specifications for a change using the openspec CLI. |
| `openspec-tasks` | When the orchestrator or user needs tasks created or updated for a change. | Create or update the implementation task checklist for a change. |
| `openspec-verify` | When the user wants to verify a completed or partially completed change. | Validate that implementation matches specs, design, and tasks. |
| `roadmap-generator` | When user asks to create, build, or generate a roadmap, change map, or implementation plan. | Generates a complete roadmap of OpenSpec changes from a project knowledge base with dependencies and rationale. |
| `sdd-apply` | When the orchestrator launches you to implement one or more tasks from a change. | Implement tasks from the change, writing actual code following specs and design. |
| `sdd-archive` | When the orchestrator launches you to archive a change after implementation and verification. | Sync delta specs to main specs and archive a completed change. |
| `sdd-design` | When the orchestrator launches you to write or update the technical design for a change. | Create a technical design document with architecture decisions and approach. |
| `sdd-explore` | When the orchestrator launches you to think through a feature, investigate the codebase, or clarify requirements. | Explore and investigate ideas before committing to a change. |
| `sdd-init` | When user wants to initialize SDD in a project, or says "sdd init", "iniciar sdd", "openspec init". | Initialize Spec-Driven Development context in any project and bootstrap persistence. |
| `sdd-onboard` | When the orchestrator launches you to onboard a user through the full SDD cycle. | Guided end-to-end walkthrough of the SDD workflow using the real codebase. |
| `sdd-propose` | When the orchestrator launches you to create or update a proposal for a change. | Create a change proposal with intent, scope, and approach. |
| `sdd-spec` | When the orchestrator launches you to write or update specs for a change. | Write specifications with requirements and scenarios for a change. |
| `sdd-tasks` | When the orchestrator launches you to create or update the task breakdown for a change. | Break down a change into an implementation task checklist. |
| `sdd-verify` | When the orchestrator launches you to verify a completed or partially completed change. | Verify that implementation matches specs, design, and tasks. |
| `skill-creator` | When user asks to create a new skill, add agent instructions, or document patterns for AI. | Creates new AI agent skills following the Agent Skills spec. |
| `skill-registry` | When user says "update skills", "skill registry", "actualizar skills", or "update registry". | Create or update the skill registry for the current project and save it to engram if available. |
| `systematic-debugging` | (No trigger metadata extracted) | Use when encountering any bug, test failure, or unexpected behavior, before proposing fixes. |
| `tailwind-design-system` | When building UIs with Tailwind CSS. | Build scalable design systems with Tailwind CSS v4, design tokens, component libraries, and responsive patterns. |
| `test-driven-development` | When implementing features or bug fixes with TDD. | Use when implementing any feature or bugfix, before writing implementation code. |
